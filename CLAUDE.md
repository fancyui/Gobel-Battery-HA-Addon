# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

Home Assistant Addon that monitors battery BMS data over serial (RS232/RS485) or TCP (Ethernet/WiFi), publishing sensor values to Home Assistant via MQTT. Supports three BMS brands: **PACE**, **JK (JKBMS)**, and **TDT**.

## Commands

There is no test suite or linter. The `CHANGELOG.md` must be updated with every version change. Version is managed in `config.yaml` (`version` field).

## Architecture

```
run.sh  ‚Üí  sensor.py  ‚Üí  bms_comm.py          (transport: serial or TCP)
                      ‚Üí  ha_mqtt.py            (MQTT publish to HA)
                      ‚Üí  ha_mqtt_jk.py         (JK-specific MQTT, composes HA_MQTT)
                      ‚Üí  pacebms_rs232.py      (PACE protocol over RS232 ‚Äî command/response)
                      ‚Üí  pacebms_rs485.py      (PACE protocol over RS485 ‚Äî multi-pack)
                      ‚Üí  jkbms_rs485.py        (JK protocol over RS485 ‚Äî passive listener)
                      ‚Üí  tdtbms_rs232.py       (TDT protocol over RS232 ‚Äî command/response)
```

- **`sensor.py`** ‚Äî Entry point. Loads `options.json` (persisted from `config.yaml` schema), initializes `BMSCommunication` + `HA_MQTT`, then dispatches to the correct BMS class based on `bms_type` + `battery_port`. Runs an infinite loop calling `publish_analog_data_mqtt()` and `publish_warning_data_mqtt()` every `data_refresh_interval` seconds.
- **`bms_comm.py`** ‚Äî Transport abstraction. `connect()` handles serial (`pySerial`) and TCP (`socket`). `send_data()` / `receive_data()` dispatch to the right transport. JK-specific methods: `receive_jkbms_passive()` for broadcast listening, `receive_jkbms_raw()` for command-based mode, `flush_jkbms_buffer()` to discard stale frames.
- **`ha_mqtt.py`** ‚Äî Paho MQTT client wrapper. Publishes HA MQTT discovery configs and state for sensors, binary sensors, events, and warnings. Entity naming: `{device_name}_{entity_id}`.
- **`ha_mqtt_jk.py`** ‚Äî JK-specific MQTT publisher. Composes (does not inherit) `HA_MQTT`. Uses different entity naming conventions (e.g., `pack_01_view_*`, `pack_01_protect_*`, `pack_01_setting_*`, `total_*`). Handles JK's binary MOS states, cell voltages, temperatures, wire resistances, setup/configuration parameters from 0x161E frame, and multi-pack aggregation.
- **BMS protocol classes** ‚Äî Each implements `publish_analog_data_mqtt()` and `publish_warning_data_mqtt()`. They handle checksum calculation, frame parsing, and call `ha_comm` (or `ha_comm_jk` for JK) to publish. The JK BMS operates in **passive mode** ‚Äî the BMS broadcasts 55AA frames continuously; the class listens and parses them. PACE and TDT use active command/response protocols.

## Key conventions

- **Isolation rule**: Modifications to any BMS module (`pacebms_rs232.py`, `jkbms_rs485.py`, `tdtbms_rs232.py`) must NOT affect the other BMS implementations.
- **MQTT topics**: `{host_name}/sensor/{device_name}_{entity_id}/config` (discovery) and `.../state` (values). JK adds `event/` and `binary_sensor/` topic prefixes.
- **Commit format**: conventional commits ‚Äî `feat:`, `fix:`, `chore:`, `docs:`.
- **Version bump**: Update `version` in `config.yaml`, add entry to `CHANGELOG.md`.
- **Config**: Runtime config lives in `/data/options.json` inside the container, generated by Home Assistant from `config.yaml` schema.
- **Pack ID**: must use pack id in ACK data to distinguish differernt packs.
